General

Clicking Processes > General displays the general parameters of processes.

Configure Process Parameters

To configure your process parameters, do the following:

  1. Make changes to each parameter by either changing the value in the field provided, or selecting or deselecting its checkbox.
  2. When you have finished making your changes, click the Save button.

Note: The process parameters that are in bold font and have an asterisk indicate that they are mandatory.

This page contains the following settings:

Field Description
Maximum delay of activity (days) If the delay of an activity is greater than this value, the process goes into an error status.
Maximum duration of active process (months)

The processes that are created earlier than the current moment minus the specified number of months are not restarted.

Note: For UI nodes, this parameter appears as read-only.
Maximum inactivity period for active process (months)

Processes that have been dormant beyond this period are not started.

Note: For UI nodes, this parameter appears as read-only.
Maximum repeat count for activities of user processes This parameter denotes the maximum allowed times an activity in a user process can be repeated. The maximum value for this field is 32000. This limit is a good safeguard against loops in the user processes due to bad metadata. Global processes are, as a rule, endless loops. As a result, their activities are never checked for a maximum number of repetitions.
Maximum number of unexpected errors before PE shuts down The process engine shuts itself down if more than the specified number of unexpected errors happens.
New processes read overlap (sec) This parameter represents the number of seconds that the new processes thread adds to the requested interval's lower limit to capture new processes that have been skipped in the previous iterations due to race conditions in the CWPROCESS table.
Participant messages id read overlap The parameter specifies the number subtracted from the participant message ID's start value when reading new participant messages. This parameter compensates for potential race conditions in the CWPPARTICIPANT table independently. However, every bucket of participant messages is updated every 1, 3, 9, 27, 81, ... minutes.
Minimum rate for reading participant messages (mins) This parameter specifies the time allotted to read participant messages. The default setting is 5 minutes.
PE requests id read overlap This parameter is the number subtracted from the request ID's start value when reading new requests from the CWPGLOBAL table. This parameter compensates for potential race conditions in the CWPGLOBAL table.
Process Start Priority Script This parameter specifies a start priority script used to define the priorities of restarting the processes. The script name is the fully qualified metadata script name. The script receives one parameter that is the current process object and must return the process restart priority, which is a number between 1 (high) and 15 (low). A non-valid return value restart the process with a priority value retrieved from the corresponding process record in the database.
Sliding exit window on critical errors (days)

Use this parameter to specify the window in days over which the number of errors parameter is to apply. This parameter works with the Maximum number of unexpected errors before PE shuts down process parameter.

Note: The process engine terminates or restarts if the number of errors encountered exceeds the specified number of errors within the window.
Sliding exit window on critical error (# of errors)

Use this parameter to specify the number of errors for the specified window (in days). The parameter works with the Maximum number of unexpected errors before PE shuts down process parameter.

Note: The process engine terminates or restarts if the number of errors encountered exceeds the specified number of errors within the window.
Load Distribution Interval (seconds) This field represents the interval that each running process engine analyzes the other running process engines and balances its load independently for each load type, trying to reach its ideal load. This field defaults to 120 seconds, and can have values between 60 and 3600 seconds.
Load Distribution Initial Interval (seconds) This field represents the initial interval that each running process engine analyzes the other running process engines and balances its load independently for each load type, trying to reach its ideal load.
Disable the last update by and last updated date modifications This field allows you to enable when activities may repeat.
Store interface process states This setting defaults to true, indicating that interface process states are automatically stored. Otherwise, change this setting to false to not store interface process states.
Family Synchronization

Selecting this checkbox ensures that the process engine runs only one activity from a process family at a time. Process family denotes the root process and all children processes.

As an example, suppose that process A invokes subprocesses B and C, and subprocess C invokes subprocess D. When subprocess B's activities run, activities from either A, B, C, or D cannot run simultaneously. These activities cannot run at the same time because the root process is A. As a result, neither process A's activities, nor its child activities (that is, B, C, and D) can run simultaneously.
Group Synchronization

Selecting this checkbox indicates that the process engine runs only one activity from a parent process and all immediate children processes at a time.

As an example, suppose that process A invokes subprocesses B and C, and subprocess C invokes subprocess D. When subprocess B's activities run, neither activities from A, nor C can run simultaneously. These activities cannot run at the same time because the parent process is A, and none of its own or immediate child activities (that is, B and C) can run simultaneously.
Order Synchronization Selecting this checkbox specifies that the process engine runs one activity at a time for any given order. As a result, processes do not compete for access to the order (for example, parent and children processes).
Global Process Synchronization Selecting this checkbox ensures that the process engine runs one global process operation for any given participant message serialization ID at a time.
