Document General Properties

The general Document properties are on the General tab.

The following table describes the fields for General tab of Document:

Field Description
Name Name of the metadata object (used in scripts; must conform to JavaScript naming conventions). Read-only after creation. To change it, use the Rename command in the popup menu by right-clicking the metadata object.
Private If checked, the metadata object is hidden from display when the metadata is packaged as Product Metadata (that is, library).
Restricted If checked, all JavaScripts contained within the object are blocked from being displayed when the metadata is packaged as Product Metadata (that is, library).
Deprecated If checked, the metadata object is designated for archival purposes only. The object becomes hidden from all metadata object selection lists.
Virtual If checked, it provides the ability to create methods in template metadata that does not have implementation, but can be used in template script and be bound to UI elements. The implementation is done in user metadata (see Virtual Field for more information).
Final Metadata elements can be marked as final, which includes any Boolean final property that is available on any metadata element. Additionally, any metadata element that has the Final property set to true can neither be extended, nor overridden.
Label Metadata object display label.
Description Description of the metadata object for documentation.
Extends Base Document to derive from. Unchangeable after Document creation.
Overrides Metadata object to override. Typically used to customize system or library metadata objects. Visible only if this Document extends from another Document. At runtime, the Document specified in Overrides field is completely replaced by this Document. A search for the overridden Document will return this Document as the result.
Dynamic Document Boolean. If enabled, the Document becomes a dynamic Document, which may have dynamic Variables, capable to be added at runtime. See topic Dynamic Document for details. Default to false.
Dynamic table name The database table name of the name-value pair (NVP) table that stores values of dynamic Variables. Only relevant when Dynamic Document is checked. See topic Dynamic Document for details.
Audit trail

If checked, the system will be triggered to record a history for any changes of the content of any runtime instance of this Document. Once it is checked, any change made to the Document will be recorded, providing the audit is requested in the Configuration application, and the Audit flag of the corresponding Document Variable is also checked. The changes are recorded into the cwaudittrail and cwfieldaudittrail database tables, which are system tables used as a history log in the database (refer to the note that follows this table).

Notes:

  • A maximum of 100 fields per document can be audited.
  • Translation fields are audited. The old and new values in the CWFIELDAUDITTRAIL table indicate the language of the change, and takes the format <language code>|<translation text> (for example, en-xx|HELLO).
  • For a dynamic document, a single operation creates one audit field record in the CWAUDITTRAIL table for both static and dynamic variables.
Attachments If checked, indicates that attachments to this Document are allowed at runtime. The attached files will be saved into the database. It is recommended that the attached files be smaller than 2 Mbytes to minimize performance degradation. Available only when Keys are defined.
Generated key

If checked, indicates that the system automatically generates the key value for this document. When checked, the Confirm Generated Key dialog appears.



If you click the Yes button, Velocity Studio tries to locate, within existing variables, a variable with the cwf.cwDocId data type to use as a key. If such a variable is not found, the key is automatically created and added to the variable list of the document. This variable is added to the key list of the document and all other keys are removed.
Timestamp sync The runtime system uses an optimistic strategy to detect situations when a Document is updated simultaneously by two or more users and/or scripts, called update conflicts. To do this, each database transaction that saves the Document will check certain database field to decide if the Document has been changed from the moment it had been read into the memory. Usually the Variable containing the namespace type System doc stamp is used for update conflict detection. However, if such a Variable does not exist and this flag  is checked, the Variable of the namespace type System last updated timestamp will be used for update conflict detection if it exists in the Document.
Keys If the Generated key checkbox is unchecked, this field defines the list of Document Variables used to uniquely identify a Document Instance. To define keys, click the Add button and select Variables from the list in the Document Tree dialog that opens. To remove an existing key, select the key and click the Remove button. When the Generated key checkbox is checked, this field contains the system-generated key and is disabled so that it can neither be changed, nor removed.
A key must be defined for the following types of Documents:
  • A Document that is mapped to a database.
  • A Document that is result of a Finder.
  • A Document that is result Document or rule Document of a Finder rule.

Note: Setting up an Audit trail for a Document will slow system performance, since extra information will be written to the database if necessary. If an audit trail is set up, to view the log content you can either use database tools or create a Finder object for table cwaudittrail, table cwfieldaudittrail or table view AuditDetails (which is a normalized view of cwaudittrail and cwfieldaudittrail). To create Finder, first import the table by selecting the Import DB Schema command from the Database menu to establish it as a Document in the metadata, then create a Finder. The column description of AuditDetails is as follows:

Column Description
DOCID Document key.
DOCMETADATATYPE Document metadata name.
UPDATEDBY The user who made the change.
LASTUPDATEDDATE The date the change was made.
OPERATIONTYPE Can be: Add, Del(elete), Upd(ate).
ATTRIBUTENAME Document Variable name.
NEWVALUE New value.
OLDVALUE Old value.
