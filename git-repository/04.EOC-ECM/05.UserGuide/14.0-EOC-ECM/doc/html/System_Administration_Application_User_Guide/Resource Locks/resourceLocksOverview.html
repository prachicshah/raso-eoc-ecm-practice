Overview

The resource locking feature provides a mechanism to place constraints onÂ the orders. With resource locking in place you can do the following:

  • Request a Resource lock. A resource can be defined as system, entity, and entity ID (for example, Siebel, Customer, 123, and so on). The response includes lock status, owner information, and the lock expiry date (lock duration, optional). The fault response includes current owner information and the lock expiry date. You can request a lock on a resource and on an order.
  • Request a lock on a resource either indefinitely or for a period of time (fixed duration, date, or time). The lock request carries requester information which is stored with the lock.
  • Request a wait for the lock to be released if the resource is already locked. It is not a real-time wait. A notification is sent on the resource release event. You can specify wait timeout and timeout handler.
  • Release the lock on a resource. The lock can only be released by the locking entity or by an administrator who provides the system, entity, and entity ID as the requester.
  • Release the lock on a timer.
  • Prevent two orders against the same subscriber, account, or location.
  • Lock a single resource requestor.
  • Prevent two users from changing the same order.
  • Queue lock requests if the resource is already locked
  • Release one lock or all for the requestor.
  • Prevent two workflows from updating the same order at the same time.
  • Notify the requestor when the lock is obtained.
  • Prevent two orders against the same service.
  • Release a lock or all locks for the requestor.
  • Remove the lock request from the queue.
  • Query for active locks and pending requests.
