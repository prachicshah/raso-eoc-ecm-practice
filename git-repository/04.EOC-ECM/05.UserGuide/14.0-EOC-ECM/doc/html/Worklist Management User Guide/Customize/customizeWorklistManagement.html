Customize Worklist Management

The Worklist Management application has been developed to facilitate customizing your worklist structure, user interface, and look and feel. The following list of events are specific to the Worklist Management application that can be used or changed to achieve a different worklist appearance and behaviour.

Use the following guidelines while customizing your application:

  1.  Publish the event to get the User Interface object to open or display it from any place in your application.
  2. Create an event handler for the event (same event and event ID) to implement a new behaviour for the event. As a result, your implementation will be used instead of the original implementation.

The following tables allow you to customize the Worklist Management application:

  • User Worklist events
  • Worklist Manager events
  • Style and icons

User Worklist events

The following table shows a list of User Worklist events, what each event returns, and its description:

User Worklist Event Returns Description
USER_WORKLIST User Interface of the User Worklist Finder

If the handler is either not implemented or empty, the User Worklist is not configured to be used for the current application. The Task(TaskNumber) application menu is not visible and the User Worklist is not accessible. If you want to use another finder instead of the original User Worklist, implement a new finder and return its User Interface from the event handler.

Any finder can behave as a worklist finder given that all proper logic and methods are implemented.

Note: To mark a new finder of type ui_common.baseFinder as a worklist set finder’s user interface, set the isWorklistFinder flag to true. Otherwise, if your finder is not using ui_common.baseFinder, implement the isWorklist method to return true.
USER_WORKLIST_RESULT_FORM User Interface that is used as the User Worklist result form To customize the User Worklist Result (for example, to use another subset of the worklist task's fields), create a new form and return it from the event handler.
USER_WORKLIST_DETAIL_FORM User Interface to be displayed once you click the task’s Detail icon from the User Worklist To customize the Detail Form (for example, to use another set of task fields), create a new form and return it from the event handler. By default, both the User Worklist and Worklist Manager use same task Detail form.
WORKLIST_TAKE_TASKS_FINDER User Interface of the Take Tasks finder To use another finder instead of the original Take Tasks, implement a new finder and return its User Interface from the event handler.
WORKLIST_TAKE_TASKS_SEARCH_FORM User Interface used as a Search Form on the Take Task finder

To customize, create a new Search Form and return the new form from the event handler.
WORKLIST_TAKE_TASKS_RESULT_FORM User Interface that is used as the Take Tasks result form To customize, create a new Result form and return the new form from the event handler.
WORKLIST_AVAILABLE_TASKS_FINDER User Interface of the Available Tasks finder To use another finder instead of the original Available Tasks, implement new finder and return its User Interface from the event handler.
WORKLIST_AVAILABLE_TASKS_SEARCH_FORM User Interface used as a Search Form on the Available Tasks finder To customize, create a new Search Form and return the new form from the event handler.
WORKLIST_AVAILABLE_TASKS_RESULT_FORM User Interface that is used as the Available Tasks result form To customize, create a new Result form and return the new form from the event handler.
WORKLIST_TASK_FORM User Interface displayed once you open the task

By default, the Task Form consists of a three-component view:

  • Task summary
  • Action menu
  • Task user interface

To customize this three-component view (for example, show task User Interface only, or change order in which components display), create a new Task Form user interface and return it from the event handler.
TASK_SUMMARY_FORM User Interface displayed as a top component in the Task Form (see WORKLIST_TASK_FORM event) To customize it (for example, include a different subset of tasks fields), create a new form and return it from the event handler.
TAKEN_TASK User Interface displayed when task of type taken task is opened

By default, it displays two fields, which provide corresponding details:

  • Taken By
  • Reason

To customize the form (for example, to include other fields), create a new form and return it from the event handler.
WORKLIST_AUTO_START_WORK None This event defines whether task is to be automatically started once opened. By default, it is set to false, meaning that task does not be automatically started. If you want to start task automatically once it is opened, implement the event handler to return true.

Note: Both the Task Menu and Task User Interface are not customizable via events as opposed to Task Summary (see the TASK_SUMMARY_FORM event). Task User Interface comes from task’s participant User Interface. User can modify each participant User Interface individually and as required directly in the metadata. Participant User Interface is fully flexible and can be implemented to display anything by making changes to its “Default” form as well as “onInit()” script.

Worklist Manager Events

The following table shows a list of User Worklist events, what each event returns, and its description:

Worklist Manager Event Returns Description
WORKLIST_MANAGER User Interface of the Worklist Manager finder If you want to use another finder instead of the original Worklist Manager, implement a new finder and return its User Interface from the event handler.
WORKLIST_MANAGER_SEARCH_FORM User Interface used as a Search Form for the Worklist Manager finder To customize (for example, to use a different set of search options), create a new Search Form and return the new form from the event handler.
WORKLIST_MANAGER_RESULT_FORM User Interface that is used as the User Worklist result form To customize Worklist Manager Result (for example, to use another subset of worklist tasks fields)`, create a new form and return it from the event handler.
WORKLIST_MANAGER_ENABLED None

This event indicates whether the Worklist Manager is enabled for the current application. By default, the Worklist Manager is always enabled if the user has one of the following privileges:

  • Worklist Administrator
  • Process Manager Administrator

You can customize the availability of Worklist Manager by including any condition and returning a result where true means that the Worklist Manager is enabled.
WORKLIST_MANAGER_DETAIL_FORM User Interface to be displayed once you click the task’s Detail icon from the Worklist Manager To customize the Detail Form (for example, to use another set of task fields), create a new form and return it from the event handler. By default, both the User Worklist and Worklist Manager use the same task Detail form.
WORKLIST_CHANGE_TASK_DETAIL_FORM User Interface to be displayed to change tasks To customize, create a new form and return it from the event handler.
WORKLIST_TASK_FINDER User Interface of the Worklist Browser finder

The finder is used from the Worklist Users finder to display all task belonging to a particular user. It is also used from the default Order User Interface to display worklist tasks for this order. Event parameters are userId and orderId, and indicate the finder's search criteria. If parameters are null, the finder returns an unfiltered result.

If you want to use another finder instead of the original Worklist Browser, implement new finder and return its User Interface from the event handler.

WORKLIST_BROWSER_RESULT_FORM

 User Interface that is used as the Worklist Browser result form To customize the Worklist Browser Result (for example, to use another subset of the worklist task’s fields), create a new form and return it from the event handler.
WORKLIST_BROWSER_DETAIL_FORM User Interface to be displayed once you click a task’s Detail icon from the Worklist Browser

To customize the Detail Form (for example, to use another set of task fields), create a new form and return it from the event handler.

By default, the Worklist Browser Detail form displays fields:

  • Priority
  • Due Date
  • Participant
  • Assigned Date
  • Start Date
  • Details

WORKLIST_ARCHIVE_FINDER

 User Interface of the Worklist Archive finder

The event's parameters are senderId and orderId, and indicate the finder's search criteria. If the parameters are null, the finder returns unfiltered results.

The Worklist Archive finder is used in the System Administration application's Process Manager finder to display the history of worklist tasks related to a particular process. It is also used from the default Order User Interface to display the history of the worklist tasks for this order.

WORKLIST_TASK_ARCHIVE_DETAIL_FORM

 User Interface to be displayed once you click the task’s Detail icon from the Worklist Archive To customize the Detail Form (for example, to use another set of task fields), create a new form and return it from the event handler.
WORKLIST_TASK_HISTORY_RESULT_FORM User Interface used as the Task History Finder result form To customize the Worklist History Result form (for example, to use another subset of the worklist task’s fields), create a new form and return it from the event handler.
WORKLIST_USERS_SEARCH_FORM User Interface used as a Search Form for the Worklist Users finder To customize (for example, to use a different set of search options), create a new Search Form and return the new form from the event handler.
WORKLIST_USERS_RESULT_FORM User Interface that is used as the Worklist Users result form To customize the Worklist Users Result (for example, to use another subset of the worklist task’s fields), create a new form and return it from the event handler.
WORKLIST_USERS_DETAIL_FORM User Interface to be displayed once you click the user’s Detail icon from the Worklist Users finder.

To customize the Detail Form (for example, to use another set of task fields), create a new form and return it from the event handler.

By default, the Worklist User’s Detail form displays the following user properties:

  • User ID
  • Total number of owned tasks, number of overdue tasks, number of alerts, and whether user is available and active.

Styles and Icons

Individual task icons and style properties used to display task records inside all finders are customizable in the application CSS file.

If the use of styles or icons, or both, need to be changed, or more styles and icons need to be added, change the following scripts:

Script Description
cwa_worklist.cwUserActiveIcon By default, active users are marked by green flags, while inactive users are marked by grey flags.
cwa_worklist.cwUserAvailabilityIcon By default, available users are marked by green flags, while unavailable users are marked by grey flags.
cwa_worklist.cwWorklistIcon

Specifies what icons are used for different task types:

  • Regular task
  • Alert
  • Taken task

Both the User Worklist and Worklist Manager use these icons.
cwa_worklist.cwWorklistStyle

Specifies styles that are used for task records on the worklist. By default, the follwing tasks have these colours:

  • Overdue tasks are red
  • Non-started tasks are bold
  • Alerts are blue
  • Taken tasks are greyed out

Both the User Worklist and Worklist Manager use these styles.
