Work With Tasks

When a process sends a task to a User Participant, these tasks are added to the global worklist queue, but not assigned to any specific user. The User Participant properties have a distribution type attribute which specifies how manual tasks should be assigned to users. This is defined by the Distribution type field on the General tab.

There are several task settings available in the user interface in a submenu titled Task. This submenu contains a list of all notificationOperations defined for each participant. Clicking a task action sends a message to the Process Engine and moves the item into the worklist archive (the user no longer sees this item in the worklist). For example, when a user B takes a task from user A, the acknowledge task appears in the user interface for user A to acknowledge this change.

Note: Errors are generated when an action cannot be completed.

Action Description
Start Work

To perform work on a task by running a task action, the task must be started. Tasks that have not started have an empty Start Date.

To start a task, select the checbox next to each task that you want to start and click the Start Work button. All selected non-started regular tasks are started. All other tasks are ignored.

Notes:

  • The Start Work action is only available if the selected work items have not been started.
  • Metadata can be set up to automatically start the task once the user opens it (that is, select the task and click the Open button in the User Worklist). By default, automatically starting a task is configured to false.
Open

This action allows you to open an already started task. To open a task, select the checkbox next to the task that you want, and then click the Open button.

See the Open Task section for more details.
Acknowledge

This task displays when a task has been taken or delegated to a user. Alerts and taken tasks must be acknowledged to be removed from the worklist.

To acknowledge a task, select the checkbox next to the tasks that you want, and then click the Acknowledge button. All selected alerts and taken tasks are closed and removed from the worklist. All other tasks are ignored.
Delegate

The Delegate option is visible to users having the Delegate Task privilege in User Profile Management. This option allows the user to reassign his tasks to another user who has not reached his maximum task capacity. Delegating tasks opens the Worklist Users finder, which displays the list of users to whom the task is allowed to be assigned. The user must make a new user selection.

A task can only be reassigned to a user having the privilege to the task’s participant. As a result, only tasks that belong to the same participant are enabled to be delegated in bulk. If the user needs to delegate tasks belonging to different participants, he must do it for each task individually.
Return (to shared queue)

The Return option is visible to users having the Return Task privilege in User Profile Management. This option allows the user to unassign his tasks (that is, return them to the available task queue).By default, a confirmation dialog appears, prompting the user to accept unassigning the selected worklist tasks.
Get Available

The Get Available option is visible to users having either the Workgroup Select or Get Available privilege only in User Profile Management. Clicking Tasks > Get Available from the menu bar allows the user to acquire one or more items from the shared task list. A user with the Workgroup Select privilege opens the Available Tasks finder, where he can select tasks to be added to his worklist. If the user does not have the Workgroup Select privilege, the next unassigned task is automatically assigned to the current user and displayed in the worklist (that is, the tasks are chosen by highest priority and earliest due date).
Take Tasks

The Take Tasks option is visible to users having the Taken On Task privilege in User Profile Management. This option allows the user to take one or more tasks belonging to another user. Clicking Tasks > Take Tasks from the menu bar brings up a finder listing tasks assigned to other users and which tasks that the current user can work on, based on task privilege.
Actions See Actions Menu for details.

Note: All worklist task actions allow for invalid objects.

Distribution of tasks are controlled by the configuration of Participants at design time. The Distribution type drop-down field on the Participants general tab contains three choices that allows you to specify how a task is routed. For details, see User Participant.

Work item actions displayed in the Actions menu support translation labels.

Actions Menu

The Actions menu lists actions that can be performed on all selected tasks. See the Task Actions section for details. An action can only be performed on a regular started task. Non-started tasks, alerts, and taken tasks display an empty action list.

Task actions are computed from the task participant metadata, meaning that each participant has its own set of actions. If all selected tasks belong to same participant type, clicking an action invokes the same action script on each task. However, if selected tasks belong to a different participant, only those actions that have same name in all of these participants can be performed. In this case, each task invokes an action strictly from corresponding participant metadata. As an example, suppose that a user selects two tasks, where task1 belongs to participant1 and task2 belongs to participant2. Both participants have actions named Complete, but each has its own action implementations. When the user clicks the Complete button, task1 invokes script participant1.complete and task2 invokes script participant2.complete.

If, however, at least one of these common-name-but-different participant actions has a confirmation script, this action is no longer available to be performed. This actions has been made to prevent multiple confirmations (one for each selected task) to run prior to performing the action. If all selected tasks belong to same participant, there is no restriction on the confirmation script and the common confirmation script runs once for all tasks.

If at least one task is non-started, or an alert or taken task, the action list is empty. When the selection consists of multiple tasks that are regular, started tasks belonging to the same participant, only those actions that are marked as Bulk Complete are listed. When the selection is a single task, all allowed actions, regardless of their Bulk Complete status, are listed.

Availability of an action for a particular task is based on both permissions and metadata configurations. As a result, when tasks belong to the same participant, each task can still have an individual subset of participant actions. When the selection consists of multiple tasks, the the list of actions is made up of intersection of actions available on each selected task.

Once an action is completed, it can either close selected tasks or leave them open, based on the action’s Keep Open flag. If the task is closed, it is removed from the worklist and worklist list refreshes. Otherwise, tasks remain open and the worklist remains unchanged. The user can work further on these tasks.
