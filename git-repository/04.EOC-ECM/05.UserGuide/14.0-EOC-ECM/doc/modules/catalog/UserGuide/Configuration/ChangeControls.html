Change Controls

The Catalog implements powerful change mechanism which guarantees the integrity of the Catalog definitions. Its task is to allow dynamic changes in the Catalog definitions without any interruption of the work of the Catalog server and client applications. From a change viewpoint the Catalog elements fall in four major groups:

  • Items – Catalog items and their associations
  • Non-versioned objects – Catalog objects that do not have versions: attribute types, association types and Catalog context type.
  • Catalog hierarchy tree elements – tree nodes, tree node children and tree node items.
  • Versioned objects – All Catalog versioned objects except items and item associations.

The change control mechanism is implemented through projects. Projects are groups of elements. The elements of the projects as well the projects themselves have a status. All catalog elements have a life span that is defined with a start and end date. The explanation of the change control mechanism below uses the term “change�? for any possible modification of the Catalog definition data. Any change can potentially have impact on the behavior of the Catalog client and server applications. However from change control view point the changes fall in two categories:

  • Adding new elements
  • Changing existing elements

The controlled changes are done by the Catalog Designer application through the Project mechanism. All changes in well designed and run Catalog environment should be controlled changes.

The controlled changes apply to any group of Catalog elements – versioned and non-versioned objects and Catalog items. The Project mechanism makes sure that the changes will become visible for all Catalog client and server applications at the same moment which is specified within the project.

How does change control work?

To make changes to a controlled environment, the following steps are taken:

  1. First a project is created. The status of the project is "definition" by default.
  2. The newly created project is Opened.
  3. Project changes are made. For example, create a new Item for this project and new attributes for this item. Associate the attributes to this item.
  4. Once the project is ready and has been approved, select the project and change the status of the project to Active. This will make the newly created item and its attributes available for production.

Notes:

  • The project needs to be activated at least 1 hour before the project effective date.
  • The start date of all changes under the project must be after the project effective date.
