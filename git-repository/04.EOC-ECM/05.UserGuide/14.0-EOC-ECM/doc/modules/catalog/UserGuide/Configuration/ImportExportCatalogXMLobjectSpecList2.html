XML Object Specifications: Item Attachment to Item Rule

When you use the XML file to import and export your catalog data, all objects have a specific format. In addition, each object`s fields have defined meaning and a required format.

The following is a list of objects, which includes a short description of the object itself and the fields that define it. The Mandatory column indicates whether the field is mandatory when importing.

  • Item attachment
  • Item attribute
  • Item charge
  • Item description
  • Item group
  • Item group target
  • Item hierarchy
  • Item image
  • Item relation
  • Item rule

Item Attachment

This entity stores large binary objects, such as Word documents, PDFs, and more, and associates them with a catalog item.

The following table describes the object's field names, their descriptions, and whether these fields are mandatory:

Field Name Description Mandatory?
itemCode This field references the item code in the item object. Yes
itemAttachmentCode This field denotes the unique code to identify the item attachment within the item. Yes
Type This field references the relation type code in the relation type object. This field defines the relation type between the item and the item attachment. No
Name This field represents the item attachment name. Yes
Status

This field indicates the state of the object, which can be one of the following:

  • definition (editable)
  • active (not editable)
 Yes
startDate

This field denotes the start date for the object in YYYY-MM-DD format.

 Yes
endDate

This field represents the optional end date for the object in YYYY-MM-DD format.

 No
Cancel

This field takes a Boolean value. If the item extends another item and this field is set to true, this attachment is considered invalid for the extended item.

 No
Label This field contains the multilingual label for the item attachment. No
createdBy This field assigns an owner to the object. By default, the owner is set to upadmin. No
createDate This field denotes the date that the object was created. By default, the create date is set as the current date and takes YYYY-MM-DDThh:mm:ss as its format. No
updatedBy This field specifies the user that last changed the object. By default, the user is set to upadmin. No
updateDate This field denotes the date that the object was last updated. By default, the update date is set to the current date and takes YYYY-MM-DDThh:mm:ss as its format. No

itemAttributeCode This field represents the unique code to identify the item attribute within the item. Yes

 Yes

Item Group Target

This entity specifies the target of the item group. It can refer to other item relations within the item or a relation type. For a relation type, all item relations within the item that satisfy the relation type criteria are eligible members of the group.

The following table describes the object's field names, their descriptions, and whether these fields are mandatory:

Field Name Description Mandatory?
itemCode This field references the item code in the item object. Yes
itemGroupCode This field references the item group code from the item group. Yes
startDate

This field denotes the start date for the object in YYYY-MM-DD format.

 Yes
itemRelationCode This field refers to the item relation code in the item relation. When specified, the item relation becomes a part of the item group. No
relationTypeCode This field refers to the relation type code in the relation type object. When specified, the targets of the item group are item relations that have the same relation type as the one specified. No
createdBy This field assigns an owner to the object. By default, the owner is set to upadmin. No
createDate This field denotes the date that the object was created. By default, the create date is set as the current date and takes YYYY-MM-DDThh:mm:ss as its format. No
updatedBy This field specifies the user that last changed the object. By default, the user is set to upadmin. No
updateDate This field denotes the date that the object was last updated. By default, the update date is set to the current date and takes YYYY-MM-DDThh:mm:ss as its format. No

Item Rule

This entity records the M:N relationship between catalog objects and rules. The rule can be either an existing catalog rule (that is, a global rule) or a private script defined within this item rule object.

The following table describes the object's field names, their descriptions, and whether these fields are mandatory:

Field Name Description Mandatory?
itemRuleCode This field represents the unique code to identify the item rule within the item. Yes
catalogObjectType

The catalog object type can be one of the following:

  • Item
  • Tax model
  • Charge type
  • Catalog hierarchy
  • Restriction
  • Item relation
 Yes
catalogObjectCode

If the type is not item, the catalog object code contains one of the following:

  • Tax model code
  • Charge type code
  • Catalog hierarchy code
  • Restriction name
  • Item relation code
 No
itemCode This field references the item code in the item object. No
ruleName Is this field is specified, it references the rule name of an existing catalog rule. Yes
startDate

This field denotes the start date for the object in YYYY-MM-DD format.

 Yes
endDate

This field represents the optional end date for the object in YYYY-MM-DD format.

 No
Status

This field indicates the state of the object, which can be one of the following:

  • definition (editable)
  • active (not editable)
 Yes
Name This field contains the multilingual label for the item rule. Yes
Type This field references the relation type code in the relation type object. This field defines the type of relation between the item and item rule. No
messageCode This field denotes the error code that displays when the rule fails. No
itemRuleScript If the item rule does not access a global rule, this field provides the catalog script that must be evaluated. No
