Projects

Projects are used to control changes to the Catalog definition and to release the changes to a live production environment in a controlled manner. The Catalog Designer application manages the Catalog definition and uses the project as a change control mechanism. Before changes are released into production, you create a project to help manage Catalog definition changes. The project name is associated to all defined Catalog changes and additions. During the design or definition stage, the project and its associated changes have a Definition status. Once the changes are ready to be moved into production, the project status is changed to Active and all the necessary information regarding the objects belonging to the project are available into production.

Projects facilitate a change control mechanism for elements under a project. The term change refers to any possible changes of the Catalog definition data that can potentially impact the behavior of the Catalog client and server applications. However, from a change control viewpoint, changes fall in two categories:

  • Adding new elements
  • Changing existing elements

Controlled changes are implemented by the project's mechanism and such changes are visible for all Catalog client and server applications at the same time. The projects are sets of changes that are committed as a single logical transaction. Users with project privileges create project and project elements can be created by any users with the appropriate privilege. Element changes under a project become effective once the project is active, and the effective date of the project and the start date of the element are reached. Make sure the project is active at least an hour before the project's effective date to ensure that the changes are realized by the change monitoring thread.

You can restrict the project data through catalog domains and positions within organizational charts. The project creator's position appears in the Owner field of the project. This field restricts the project data to users who have the same position in the organizational chart or higher. Once you have specified an owner, it can neither be changed, nor transferred.

Projects go through a development stage where project changes are created and approved. After the development stage, the project must either be committed (for example, an active project) or cancelled. The project has a number of different system stages:

  • Definition
  • Development
  • Conception
  • Evaluation
  • Testing
  • Launch Product
  • Active

When a project is first created, the project status defaults to Definition. The Development, Conception, Evaluation, Testing, and Launch Product statuses are only for information purposes and are used in the development stage. It is in this stage that the project's effective data can be changed at any time, but it can be set not earlier than one hour past the moment of the change. Once a project has an Active status, all changes associated with that project are available to production. Further changes can neither be made to that project, nor can that project be deleted. A new project must be created to make additional catalog changes.

You can perform the following actions with a project:

  • Create a new project
  • Open an existing project
  • Delete a project
  • Move part of a project to another project
  • Add elements to a project
  • Activate a project

To import a project, see Import Catalog for details.
