Add Elements to a Project

New elements can be freely created under a project and projects can freely create new versions of any element with the following restrictions:

  • Only one version of an element can be created in a project and belongs to this project. Once you have created this element, it cannot be moved to other projects.
  • Catalog objects are read-only if the current project is different from the project that the object belongs to.
  • Any change of an association of an object (versioned or not) is considered a change of this object.
  • Only one project in development stage can change a given Catalog object.

The new elements created in the project have a default start date that is equal to the system date when the object was created. The start date of objects under the project have to be after the effective date of the project. Otherwise, the system will produce a validation error when the user tries to activate the project. The example below shows the Catalog behavior for a project with an existing item object change and when the changes are picked up by the change monitoring thread.

Date Action
Mar 1, 2010 Project created with effective start date June 1, 2010
Mar 4, 2010 New association added to ITEM
Mar 6, 2010 Another association added to ITEM with a start date of June 1st.
Apr 25, 2010 Project is committed
June 3, 2010 Item Start Date

The result of the request for the ITEM in different time moments for a request date past the project's effective date is given below. The old content of ITEM is marked as OLD, the new as NEW.

Date Action
Feb 15, 2010 OLD
Mar 2 2010 OLD
Mar 5, 2010 OLD
Apr 5, 2010 OLD
May 5, 2010 OLD
June 3, 2010 NEW

The project is committed on April 25. However, the effective date of the project is not until June 1st. The project change monitoring thread will notice this change and process it. But the new item specifications will not become active in production until the API request date is on or after June 1.
